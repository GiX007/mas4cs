# MAS4CS: A Multi-Agent System for Customer Service Automation

A modular, hierarchical multi-agent system for end-to-end customer service automation using LangGraph and MultiWOZ 2.2.

---

---

## Why This Project

A perfect customer service system understands the user's goal, tracks all constraints without losing context, enforces business rules before acting, avoids hallucinations, handles multi-domain conversations, and completes the task reliably from start to finish.

Most single-LLM approaches fail at one or more of these requirements under real-world complexity. MAS4CS investigates whether a structured, hierarchical multi-agent architecture rather than a larger model can close this gap reliably.

## How It Works

MAS4CS is built on one core question: does **architectural structure** alone improve reliability, over single large models?

Five specialized agents are connected in a deterministic LangGraph workflow, each with a single responsibility. A shared `AgentState` flows through the graph. Every agent reads from it and writes back to it.

```
User Message → Triage → Policy → Action → Supervisor → Memory
                                    ↑           |
                                    |__(retry)__|
```

- **Triage** detects domain, intent, and extracts slots
- **Policy** enforces hard rules (no booking without required slots)
- **Action** generates the response, queries the DB for entity grounding
- **Supervisor** validates the response and triggers a retry if needed
- **Memory** saves the approved turn to conversation history

The Supervisor validates before Memory saves. If validation fails and fewer than 2 attempts have been made, the workflow loops back to Action for self-correction. 
Memory only saves after Supervisor approves or max retries are exhausted.

**A note on Agents:** In MAS4CS, agents are structured LLM nodes with distinct roles and not as autonomous tool-using agents in the classical sense. 
No external tools or APIs are called beyond the local MultiWOZ DB files. The goal is to study whether architectural structure alone improves reliability.

Full details in `docs/project_docs/agents.md`.

## Experiments

Three experiments isolate the effect of architecture and model specialization:

- **Experiment 1:** Single-agent baseline: one LLM, one prompt, no architecture
- **Experiment 2:** MAS graph: same models, structured multi-agent workflow (homogeneous + heterogeneous configs)
- **Experiment 3:** Fine-tuned MAS: role-specialized open-source models via LoRA (EuroHPC)

Full details in `docs/project_docs/experiments.md`.

## Evaluation

Metrics across three levels (turn, dialogue, dataset):

**Custom metrics:**
- Routing & Intent: Intent Accuracy, Domain Accuracy
- Slot Tracking: Slot Accuracy, JGA, Hallucination Rate
- Action: Action-Type Accuracy
- Policy & Booking Task: Policy Compliance, Task Completion, System Correctness
- Cost per dialogue (USD), Average latency per turn (seconds)

**Official metrics (Tomiinek evaluator):**
- Inform Rate, Success Rate

## Development Roadmap 

Exact steps to reproduce the MAS4CS pipeline from data loading to evaluation.

- **Step 1: Data**
  - `load_multiwoz()` — loads or downloads dataset, caches to `src/hf_cache/`. Called in `main.py`
  - `python -m tests.test_loader` — validates dataset loading and split sizes (HF + official GitHub)
  - `run_preprocessing_pipeline()` — applies preprocessing and caches to `dataset/mw22_filtered.json`. Called in `main.py`
  - `python -m tests.test_preprocess` — validates transformation and slot normalization on one sample
  - `python -m scripts.explore_dataset` — full exploration pipeline of HF dataset. Saves 8 inspection files to `docs/dataset_inspection/` and 4 schema/data files to `dataset/`
  - `python -m scripts.explore_dataset_v2` — full exploration of official GitHub dataset structure, DB files, and conversation examples. Saves output to `docs/dataset_inspection/mw22_github_exploration.txt`
  - `src/data/load_dataset_v2.py` — loads and filters official MultiWOZ 2.2 from GitHub (all splits, hotel + restaurant only)
  - `docs/project_docs/dataset.md` — full dataset documentation: structure, feature selection, HF vs GitHub differences, preprocessing, and processed format examples

- **Step 2: Models**
  - `docs/project_docs/models.md` - model arsenal reference: open-source and commercial models, stack configurations, fine-tuning strategy
  - `src/models.py` — unified LLM interface (OpenAI, Anthropic, Unsloth)
  - `python -m tests.test_models` — tests API and local model calls. Saves results to `docs/model_inspection/` (2 JSON + 2 TXT files per run)
  - `python -m scripts.explore_models` — raw output exploration across all available models

- **Step 3: Agents**
  - `docs/project_docs/agents.md` — full agent documentation: fundamentals, workflow architecture, tool design, agent roles, and shared state reference
  - `python -m scripts.explore_agents` — demonstrates agent architecture progression: raw LLM → tool-calling agent → ReAct loop. Saves output to `docs/agents_inspection/`
  - `src/core/state.py` — `AgentState` TypedDict and `initialize_state()` factory
  - `src/agents/triage.py`, `policy.py`, `action.py`, `memory.py`, `supervisor.py` — individual agents
  - `src/tools/db_tools.py` — `find_entity` and `book_entity` for DB-grounded responses
  - `python -m tests.test_agents` — unit tests for all individual agents with real LLM calls
  - `src/core/workflow.py` — LangGraph workflow graph with retry loop. Called internally by experiments and `main.py`
  - `python -m tests.test_workflow` — tests full pipeline on one real MultiWOZ turn. Saves workflow graph images to `docs/images/`

- **Step 4: Evaluation**
  - `docs/evaluation.md` — full evaluation documentation: metrics, formulas, examples
  - `src/evaluation/evaluation_metrics.py` — all custom objective metric functions
  - `src/evaluation/llm_judge.py` — LLM-as-judge infrastructure
  - `src/evaluation/evaluator.py` — `DialogueEvaluator` and `DatasetEvaluator`
  - `python -m tests.test_evals` — tests all metrics with examples. Saves output to `docs/evals_inspection/objective_and_judge_metrics.txt`. Also, see `metrics_quick_reference.txt`
  - `python -m tests.test_evaluator` — integration tests for `DialogueEvaluator` and `DatasetEvaluator`. Saves output to `docs/evals_inspection/evaluator_test_results.txt`
  - `python -m tests.test_tomiinek_evaluator` — integration tests for Tomiinek Evaluator. Saves output to `docs/evals_inspection/tomiinek_evaluator_tests.txt`

- **Step 5: Experiments**
  - `docs/project_docs/experiments.md` — experiment setup, model configs, architecture configurations, results structure
  - `src/experiments/config.py` — single source of truth for all experiment parameters
  - `python -m main` — runs all configured experiments
  - Results saved to `results/` (3 JSON files per run + `leaderboard.txt`), overall and per-domain
  - Results of 1 dialogue-run to `results/debug_traces/`. Full per-run execution logs with prompts sent to LLMs, raw model responses, and turn-level evaluation snapshots
  - Results analysis saved to `docs/project_docs/results.md`

- **Step 6: References**
  - `docs/project_docs/references.md` — academic and technical work related to this project
